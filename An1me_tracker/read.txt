v6.6.6 — 2026-05-25
====================

WHAT CHANGED
------------

1) Episode-count flap FIXED (3432↔3440 oscillation, 52d↔53d watch-time)
   Root cause: legacy `autoRepaired:true` episodes lived in some devices'
   chrome.storage.local.animeData. The popup's loadAndSyncData stripped
   them via removeAutoRepairedEpisodes, but the BG syncToFirebase /
   _doApplyCloudUpdate paths did NOT — so the BG kept re-merging the
   un-stripped local copy with the smaller cloud doc and pushing the
   8 episodes back. Popup then stripped again. Endless flap.

   Fix in src/common/merge-utils.js: drop autoRepaired:true episodes
   inside mergeAnimeData itself (the choke point all sync paths go
   through). Also handle single-sided entries (where only cloud or
   only local has the slug) via the new stripAutoRepairedEpisodesFromMap
   helper, exported on AnimeTrackerMergeUtils.

   Plus: applied the helper in the no-cloud-doc fallback paths of
   BG syncToFirebase + popup performCloudSave + content-script
   pushFullDirect, so even the very first sign-in push doesn't seed
   cloud with autoRepaired flags.

   After this update, autoRepaired episodes get permanently cleaned
   from both local AND cloud after a single merge cycle. Total
   episode count and watch-time will converge to the lower (correct)
   value across all devices and stop flapping.

v6.6.5 — 2026-05-25
====================

WHAT CHANGED
------------

1) AniList legacy auto-push gate (FIX)
   v6.6.3-6.6.4 only pushed local→cloud when local had an `updatedAt`
   stamp. But existing desktop installs had `anilist_auth` set BEFORE
   the sync feature shipped → no stamp → never auto-pushed. Mobile
   never saw the token in cloud and kept showing "sign in on desktop".
   Fix in src/popup/firebase-sync.js loadAndSyncData reconcile:
   push when (a) cloud has no token (regardless of stamps) OR
   (b) both stamped + local stamp newer.

2) Mobile boot safety net (NEW)
   Fixes the window where desktop pushes the token but mobile's SW
   serves a stale cached doc / waits out the 3-min poll gate. When
   mobile-like browser + signed in + AniList disconnected:
   - Popup sends WAKE_AND_POLL_CLOUD_FORCE on boot
   - BG bypasses the 3-min gate AND the cache freshness check
   - One forced Firestore read, then applyCloudUpdate writes the
     token locally → AniList card flips to connected on its own
   Fires at most once per popup session, only when needed.

3) New BG message: WAKE_AND_POLL_CLOUD_FORCE
   - pollCloudData now accepts { force: true } to bypass both gates.
   - Existing WAKE_AND_POLL_CLOUD (from content scripts) unchanged.

FILES TOUCHED

- manifest.json                  6.6.4 → 6.6.5
- README.md                      version bump
- background.js                  pollCloudData force option +
                                 WAKE_AND_POLL_CLOUD_FORCE handler
- src/popup/firebase-sync.js     legacy push gate fix + diagnostic log
- src/popup/anilist-api.js       mobile boot safety net

USAGE NOTES
-----------
- Desktop user must open the popup ONCE after updating to v6.6.5+
  so the auto-push fires and seeds the cloud.

  within seconds (not 5 minutes).

v6.6.4 — 2026-05-25
====================

WHAT CHANGED
------------

1) Sign-out persistence bug FIXED (mobile + desktop)
   Root cause: refresh-token failure on flaky/cold-boot networks would
   wipe the session even when the failure was transient (e.g. mobile
   browser woke the SW before network was reachable).
   Fix: classify failures as PERMANENT vs TRANSIENT. Only permanent
   (revoked refresh token, disabled account) signs out. Transient
   keeps existing token if still valid; otherwise keeps session for
   the next retry. Applied in 3 places:
     - src/popup/firebase-lib.js   (popup init + getIdToken)
     - background.js               (BG getFirebaseToken)
     - src/content/cloud-sync.js   (CS getValidToken)

2) Idle Firebase reads cut to ~zero
   - background.js pollCloudData → skips the Firestore read when the
     SW cache is still fresh (within _BG_CLOUD_TTL=5min). Previously
     every consumer-connected wake burned a read.
   - background.js syncProgressOnly + syncToFirebase already seed the
     cache (from v6.6.3); kept.
   - src/content/cloud-sync.js: 4 PATCH success paths
     (pushProgressDirect, pushFullDirect, 2× keepalive-on-unload)
     now SEED the BG cache via UPDATE_BG_CLOUD_DOC_PARTIAL instead of
     invalidating. New BG handler overlays partial fields; uid-verified.

3) Idle Firebase writes cut to zero
   - src/popup/firebase-sync.js performCloudSave: new idempotency gate
     compares mergedAnime/Progress/Deleted/Group/Goals/Badges vs the
     cached cloud doc. If all match AND email is already synced → skip
     the PATCH. Logs "idempotent-skip" (throttled 5s).

4) New BG message protocol additions
   - UPDATE_BG_CLOUD_DOC_PARTIAL { uid, partial } → overlay onto cache

NET EFFECT
----------
Opening popup with no changes across devices: 0R + 0W against Firestore.
Extension reload on mobile: session preserved (no sign-out).
